Chat Flow

End-to-end journey from user input to AI response

Session Initialization

When the ChatWidget mounts, it must establish a session before any messages can be sent.

Check Visibility

GET /api/chat/visibility — returns { chatbotEnabled: boolean }. If disabled, the widget stays hidden and no session is created.

Look for Existing Session

Checks sessionStorage["chatWidgetSessionId"]. If found, the stored UUID is reused (sessions survive page refreshes within the same tab).

Create New Session (if needed)

POST /api/chat/session — server generates crypto.randomUUID(), upserts a ChatSession document in MongoDB, and returns the sessionId. A singleton promise prevents duplicate creation on concurrent renders.

Why sessionStorage? Each browser tab gets its own independent chat session. Closing the tab naturally clears the session. This avoids stale conversations leaking across tabs.

Backend Controller Steps (chat.controller.js → askQuestion)

Input Validation

sessionId: string, max 100 chars. question: string, 1–2000 chars. The frontend-side .escape() was intentionally removed to avoid mangling legitimate user input — the backend sanitizer handles XSS.

Chatbot Enabled Check

Reads chatbotEnabled from AdminSettings. Returns HTTP 503 if disabled.

Ensure Session

Upserts the session document — if the sessionId doesn't exist, it creates one. This handles edge cases where sessionStorage has a stale ID.

Guard Check #1 — isBlockedQuery()

Tests against offensive, confidential, and prompt-injection patterns. If blocked, returns a safe fallback message and logs with status: "fallback".

Guard Check #2 — isOutOfScope()

Tests against out-of-scope patterns (code generation, recipes, sports, etc.). Returns a polite redirection message.

Retrieve History

Loads conversation history from the ChatSession's messages[] array. This gives the AI context about the ongoing conversation.

Run RAG

Calls runRAG(question, history) which embeds, retrieves, and generates the answer. See the RAG Pipeline section for details.

Persist & Respond

Adds both user & assistant messages to the session. Creates a ChatLog with full metadata (timing, sources count, model info, IP, user-agent). Emits analytics:updated and information:updated via Socket.IO. Returns { answer, sources }.

Session Recovery: If the backend returns 404 ("session not found"), the frontend auto-retries once by clearing sessionStorage and creating a fresh session.

Quick Action Buttons

Three preset buttons appear when the conversation is empty. Clicking one sends it as if the user typed it:

"What jobs are available?"
"How do I apply?"
"What are the benefits?"

These are passed directly to sendMessage(action) — the same function used for typed messages. No special handling needed on the backend.

RAG Pipeline

Retrieval-Augmented Generation: Embed → Retrieve → Generate

End-to-End RAG Flow

User Question"What internships are available?"
→
Embedtext-embedding-3-small
→
RetrieveElasticsearch kNN
→
GenerateGPT-4o-mini
→
AnswerJSON response

Step 1 — Embed the Question

The user's question is converted into a 1536-dimensional vector using Azure OpenAI's text-embedding-3-small model via getEmbeddingClient().embeddings.create(). 

// embedding.service.js
const response = await getEmbeddingClient().embeddings.create({
  model: AZURE_EMBEDDING_DEPLOYMENT,
  input: text
});
return response.data[0].embedding; // Float[1536]

This vector captures the semantic meaning of the question and will be compared against all stored content vectors.

Step 2 — Retrieve Relevant Chunks

The embedding vector is sent to Elasticsearch's kNN (k-Nearest Neighbors) search, which finds the most semantically similar content chunks. 

// retrieval.service.js → searchRelevantChunks()
const result = await esClient.search({
  index: "unido_careers_index",
  knn: {
    field: "embedding",
    query_vector: embedding,
    k: TOP_K,              // default 5
    num_candidates: 50     // broader initial scan
  }
});
// Filter by MIN_RELEVANCE_SCORE (default 0.75)
ParameterDefaultEnv VariablePurpose
TOP_K5RAG_TOP_KMaximum chunks returned
MIN_RELEVANCE_SCORE0.75RAG_MIN_SCORECosine similarity threshold — rejects low-confidence matches
num_candidates50—Elasticsearch evaluates this many candidates before picking top K
No relevant results? If no chunks pass the minimum score threshold, the system returns "I do not have enough information to answer that question" without calling GPT — saving cost and preventing hallucination.

Step 3 — Generate the Answer

The retrieved chunks are formatted as context and sent alongside the question to GPT-4o-mini.

// generation.service.js
const messages = [
  { role: "system", content: systemPrompt },
  ...normalizedHistory,
  {
    role: "user",
    content: `QUESTION: ${question}\n\nCONTEXT:\n${contextText}`
  }
];

const response = await getChatClient().chat.completions.create({
  model: AZURE_OPENAI_DEPLOYMENT,  // gpt-4o-mini
  messages,
  temperature: 0.1  // near-deterministic
});

System Prompt (configurable via Admin UI)

The system prompt is loaded from AdminSettings.systemPrompt. Default: 

"You are a UNIDO Careers Assistant. Answer ONLY from the provided CONTEXT.
If the context does not contain enough information, say you don't have
enough information. Do not make up answers."

Why temperature 0.1?

Near-zero temperature ensures the AI stays faithful to the retrieved context. For a careers information chatbot, factual accuracy is critical — creative or speculative answers could mislead job applicants.

Six scrapers extract all content from the UNIDO Careers website using Puppeteer (headless Chrome):

  • Returns a Cheerio $ instance for further DOM querying by the scrapers

Phase 4 — Embedding & Indexing

Ensure Index

Creates the unido_careers_index in Elasticsearch with dense_vector mapping (1536 dimensions, cosine similarity) if it doesn't exist.

Clear Existing Data

deleteByQuery({ match_all: {} }) removes all documents from the index to ensure a clean re-index.

Generate Embeddings

For each chunk, withEmbeddings() calls Azure OpenAI to generate a 1536-dimensional vector. Processed one-by-one to respect rate limits.

Bulk Index

All chunks (with embeddings) are indexed using Elasticsearch's bulk API for efficient writes.

Save Backup

JSON files saved to local data/ and chunks/ folders with date-prefixed filenames (e.g., 180226_unido_faqs.json).

Cron Schedule: When ENABLE_CRON=true, the pipeline runs daily at midnight UTC via node-cron schedule "0 0 * * *". Timezone is configurable via CRON_TIMEZONE.

Authentication Flow

Admin login, JWT verification, and route protection

Login Sequence

→
JWT SignHS256 · 25min expiry
→
localStorage"adminToken"

Backend Details (adminAuth.service.js)

Find Active User

Queries AdminUser.findOne({ email, isActive: true })

Constant-Time Defense

If user not found, hashes a dummy password anyway to prevent timing-based user enumeration attacks.

Password Verification

bcrypt.compare() with 12 salt rounds (OWASP recommended).

Token Generation

Signs JWT with { id, role: "admin", iat }, algorithm HS256, expiry from JWT_EXPIRES (default 25m). Secret must be 32+ characters in production.

Token Verification Middleware

Every admin API request passes through requireAdminAuth:

Extract Token

Reads Authorization: Bearer <token> header.

Format Validation

Checks 3-part JWT format (header.payload.signature). Rejects malformed tokens before decoding.

JWT Verify

jwt.verify() with algorithms: ["HS256"] and maxAge: "30m".

Role Check

Verifies decoded.role === "admin". Returns 403 Forbidden with IP logged if role mismatch.

Security: Expired tokens get a specific TokenExpiredError response (401). Other verification failures get a generic 401. This prevents information leakage about why a token was rejected.

Client-Side Route Protection (PrivateRoute.jsx)

A React wrapper component that guards admin routes:

  1. Reads token from localStorage.getItem("adminToken")
  2. Parses the JWT payload via base64 decode (no verification — server handles that)
  3. Checks exp timestamp with a 30-second buffer for clock drift
  4. Validates role === "admin"
  5. If invalid → removes token → redirects to /login
  6. If valid → renders the protected children
Defense in depth: Client-side checks are for UX only. Every API call is independently verified by the server middleware. A tampered client-side token can't access real data.

Real-time Communication

Socket.IO events for live updates

Socket.IO Architecture

Socket.IO runs on the same Express server. Clients connect via WebSocket with polling fallback. Admin clients include their JWT in the auth handshake for elevated access.

Connection Lifecycle

Client Connects

Singleton pattern via getSocket(). Transports: ["websocket", "polling"]. Admin clients send auth.token.

Server Auth Middleware

If auth.token is present, verifies JWT → sets socket.isAdmin = true. Invalid/missing token allows connection but marks isAdmin = false.

Ready Event

Server emits socket:ready with timestamp on successful connection.

Idle Disconnect

30-minute timeout — idle connections are automatically closed to free resources.

Multi-layer security posture protecting UNIDO data, users, and infrastructure

The chatbot employs a defense-in-depth strategy with seven distinct security layers, each designed to catch what the previous layer might miss:

1. Data Protection & Privacy

MeasureImplementationWhy It Matters
No PII Storage The chatbot does not collect or store any personally identifiable information. Chat sessions are anonymous (UUID-based) and identified only by IP for rate limiting. Ensures GDPR and UN data protection compliance — no user data to breach
PII Query Blocking Guard service blocks questions containing confidential patterns: "staff salary", "passport", "ssn", "credit card", "medical record", "employee data" Prevents users from attempting to extract sensitive organizational information via the AI
Session Isolation Each browser tab gets a unique sessionStorage-based session. Sessions cannot cross-reference each other. Closing the tab destroys the session client-side. Prevents conversation leakage between users or tabs
Response Grounding AI answers ONLY from retrieved UNIDO content (RAG). Temperature 0.1 prevents hallucination. System prompt explicitly forbids made-up answers. Eliminates risk of the AI revealing data it shouldn't know or generating false information
No External Data Leakage CSP restricts connect-src to 'self'. No third-party analytics, tracking pixels, or external API calls from the frontend. User interactions never leave the UNIDO domain boundary

2. Authentication & Access Control

MeasureImplementationStandard
Password Hashing bcrypt with 12 salt rounds — each hash takes ~250ms, making brute-force computationally expensive OWASP recommended minimum: 10 rounds
Timing Attack Defense Login always hashes a dummy password even when user not found — constant response time regardless of user existence OWASP Authentication Cheat Sheet
JWT Security Algorithm locked to HS256 (prevents "none" algorithm attack). 25-minute expiry. 32+ character secret enforced in production. maxAge: "30m" server-side verification. RFC 7519, OWASP JWT Cheat Sheet
Auth Rate Limiting 5 login attempts per minute per IP. Does NOT skip successful requests — prevents credential stuffing even with valid credentials. OWASP Brute Force Prevention
Password Policy Minimum 8 characters (API validation), 12+ characters recommended and warned on bootstrap. Max 128 characters to prevent bcrypt DoS. NIST SP 800-63B
Role-Based Access JWT contains role: "admin". Middleware verifies role on every request. Non-admin tokens receive 403 with IP logged. Principle of Least Privilege

3. Network & Transport Security

MeasureImplementation
HTTPS Enforcement HSTS with 1-year max-age, includeSubDomains, preload. Automatic HTTP→HTTPS redirect in production via x-forwarded-proto check.
CORS Strict Origin Production allows only https://careers.unido.org. Requests without Origin header blocked in production (prevents direct API abuse).
Frame Protection X-Frame-Options: DENY and frame-src: 'none' — prevents clickjacking by forbidding embedding in iframes.
Content Sniffing X-Content-Type-Options: nosniff — prevents browsers from MIME-sniffing responses away from declared content type.
Permissions Policy Disables camera, microphone, and geolocation APIs — these are not needed and represent unnecessary attack surface.
Cross-Origin Isolation Cross-Origin-Embedder-Policy, Opener-Policy, and Resource-Policy all set to same-origin.
Fingerprint Minimization X-Powered-By header removed. DNS prefetch disabled. Cache-Control: no-store prevents sensitive data caching.

4. Input Security & Injection Prevention

Attack VectorDefenseLayer
Cross-Site Scripting (XSS) Input sanitizer blocks <script>, javascript:, on*= patterns. CSP blocks inline scripts. React's default escaping provides additional defense. Middleware + CSP + Framework
NoSQL Injection Dedicated noSQLInjectionGuard rejects any request body or query with keys starting with $ (MongoDB operators like $gt, $ne, $or). Middleware
Template Injection Input sanitizer blocks ${} and {{}} patterns — prevents server-side and client-side template injection. Middleware
ReDoS (Regex DoS) Search input in the admin table is escaped via escapeRegExp() before being used in MongoDB regex queries. Service layer
Prompt Injection Guard service blocks: "ignore previous instructions", "you are now", "system:", "DAN jailbreak", "[INST]", "<<SYS>>", "forget everything", and 10+ other patterns. Application layer
Payload Overflow payloadGuard rejects bodies > 2 MB. Input validation enforces max lengths on all fields (question: 2000, sessionId: 100, systemPrompt: 500, search: 200). Middleware + Validation
HTTP Method Abuse methodGuard allows only GET, POST, PUT, DELETE, OPTIONS, HEAD — blocks TRACE, CONNECT, PATCH, etc. Middleware

5. Monitoring & Audit Trail

CapabilityImplementation
Request Tracing Every request gets a unique requestId ({timestamp}-{counter}). All logs reference this ID for end-to-end tracing.
Audit Logging auditLogger middleware records: request ID, HTTP method, path, status code, response time, IP address, and user-agent for every request.
Error Alerting Responses with status ≥ 400 are logged at WARN level. Slow requests (> 5 seconds) are flagged separately.
Chat Audit Every Q&A interaction is stored in ChatLog with: question, answer, status, guard reason, response timing, source count, model used, IP, and user-agent.
Admin Activity All admin API calls go through JWT middleware — the decoded token provides admin identity for every action. Failed auth attempts log IP addresses.
Security Events Blocked queries (offensive, injection, PII) are logged with status: "fallback" and specific guardReason for security review.

6. AI-Specific Security

RiskMitigation
Hallucination RAG-only architecture with temperature 0.1. System prompt: "Answer ONLY from provided CONTEXT." If no relevant chunks found (score < 0.75), returns "I do not have enough information" without calling GPT.
Data Poisoning Content is scraped only from official UNIDO domains (careers.unido.org, unido.org). No user-submitted content enters the knowledge base. Full re-index clears stale data.
Prompt Injection Four-layer guard service filters input before it reaches the AI. Patterns block system prompt override attempts, DAN jailbreaks, and instruction manipulation.
Scope Creep Out-of-scope pattern detection redirects users asking about code, recipes, sports, stocks, etc. The AI is scoped strictly to UNIDO careers content.
Token Abuse Chat rate limit (100/min per IP) + input max length (2000 chars) + speed limiter (500ms delay after 50 requests) prevents excessive API/token usage.

🤖 Azure OpenAI API

Critical

What it is: Microsoft’s enterprise-grade AI service that hosts large language models (LLMs) such as GPT-4o-mini and embedding models like text-embedding-3-small.

Why we need it: Azure OpenAI API powers the chatbot’s core intelligence. It is responsible for two key capabilities:

  • Understanding questions — The embedding model converts every user question into a 1536-dimensional vector that captures its semantic meaning, enabling the system to find the most relevant career content even when the user’s phrasing doesn’t match the exact words on the website.
  • Generating responses — The GPT-4o-mini chat model reads the retrieved career content and composes accurate, human-like answers grounded strictly in UNIDO’s official information. Temperature is set to 0.1 to minimize hallucination.

🔍 Elasticsearch (Vector Database)

Critical

What it is: A distributed search and analytics engine that supports k-nearest-neighbor (kNN) vector search, acting as the project’s vector database.

Why we need it: Elasticsearch enables fast semantic search — the backbone of the RAG (Retrieval-Augmented Generation) pipeline. When a user asks a question:

  1. The question is converted into a vector embedding via Azure OpenAI.
  2. Elasticsearch performs a kNN search across all stored content vectors to find the most semantically relevant chunks of career information.
  3. The top-matching chunks are passed to GPT-4o-mini as context so it can generate a grounded, factual answer.

Without Elasticsearch, the chatbot would have no way to locate relevant information from the hundreds of scraped pages and would be unable to answer any question.

🗃 Azure Blob Storage

Critical

What it is: Microsoft Azure’s scalable object storage service for unstructured data such as JSON files, HTML snapshots, and backups.

Why we need it: Azure Blob Storage serves as the persistent data layer between the scraping pipeline and the indexing pipeline:

  • Raw scraped content — After the scraper collects job listings, FAQs, category pages, and homepage content from the UNIDO careers website, the raw JSON data is uploaded to Blob Storage. This creates a reliable, versioned archive of all scraped data.
  • Chunked content — The chunker processes raw data into smaller, semantically meaningful pieces (chunks). These chunks are also stored in Blob Storage before being embedded and indexed into Elasticsearch.
  • Backup & re-indexing — If Elasticsearch needs to be rebuilt (e.g., after a schema change or data corruption), the system can re-read all chunks from Blob Storage and re-index without having to re-scrape the entire website.

🛡 Azure AI Content Safety / Guardrails

Important

What it is: Azure AI Content Safety is a cloud service that detects harmful or inappropriate content across text and images, providing severity scores across categories like hate, violence, sexual content, and self-harm.

Why we need it: While the chatbot already includes a multi-layer guard service (offensive patterns, prompt injection detection, PII blocking, and out-of-scope filtering), Azure AI Content Safety adds an enterprise-grade, continuously-updated safety layer:

  • Response filtering — Before any AI-generated response is sent to the user, it can be scanned by Content Safety to ensure it does not contain inappropriate, harmful, or non-compliant content — even if the underlying model produces something unexpected.
  • Input screening — User questions can be pre-screened to detect abusive or manipulative inputs that might bypass regex-based pattern matching.
  • Compliance assurance — Provides auditable safety scores and categories for every interaction, helping UNIDO demonstrate responsible AI usage in line with UN digital governance standards.
  • Adaptive protection — Unlike static regex patterns, Azure AI Content Safety uses ML models that are continuously updated by Microsoft to catch emerging threats and adversarial techniques.
